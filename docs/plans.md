# Design Desk Implementation Plan

## Summary

Design Desk is a local-first, TypeScript + React + Vite + Tailwind design editor built for a long-horizon Codex run. The app must impress non-engineers in a live demo and engineers in code review: polished editor UI, multiplayer presence, replay, version history, deterministic export, strong types, and reliable tests.

This file is the milestone contract. `docs/prompt.md` is the frozen source specification. `docs/implement.md` is the execution loop. `docs/documentation.md` is the live ledger.

The user preference for this repo is that the canonical plan file lives at `docs/plans.md`.

## Decisions

- Scope: build the full `docs/prompt.md` contract, not a reduced MVP.
- Codex run mode: local `/goal`, auto-continue milestone by milestone with verified commits.
- Milestone count: 21 focused milestones.
- Implementation order: deterministic spine first, then editor capabilities, collaboration, replay, export, and polish.
- Rendering: SVG is the authoritative editable surface. Canvas is reserved for previews, thumbnails, minimap, replay preview, or future performance experiments.
- State: a custom typed deterministic operations journal drives undo, redo, replay, snapshots, sync, and export.
- Collaboration: local Node WebSocket server assigns canonical sequence numbers and broadcasts ordered ops.
- Persistence: local Node server writes sessions and designs to a gitignored `data/` directory; tracked demo files seed the starter projects.
- IDs: seeded deterministic ID factory with allocated IDs persisted in ops.
- Demo direction: professional editor UI with AI builder suite sample projects.
- Assets: tracked local generated assets only; no remote demo image dependencies.
- Export: deterministic React + Tailwind component package generated by CLI.
- Verification: lint, typecheck, unit tests, deterministic snapshots, build, and targeted Playwright smoke tests once UI flows exist.
- Reasoning profile: use High for normal milestones and Extra High for architecture, sync, replay, export, and difficult debugging.

## Architecture Overview

### Scene Graph Data Model

The scene graph is a normalized document with stable IDs, root frames, parent-child relationships, z-order arrays, style tokens, component definitions, component instances, comments, prototype links, snapshots, and an append-only ops journal.

Core node kinds are Frame, Group, Rectangle, Ellipse, Line, Text, Image, Button, Icon, ChartPlaceholder, ComponentRoot, and ComponentInstance. Geometry uses deterministic floating-point normalization before serialization. Parent-child ordering is explicit and stable.

Likely modules:

- `src/model/scene.ts`
- `src/model/node.ts`
- `src/model/styles.ts`
- `src/model/components.ts`
- `src/model/comments.ts`
- `src/model/prototype.ts`

### Selection And Transform System

Selection is stored as ordered selected IDs plus an active anchor ID. Hit-testing resolves from topmost visible unlocked node to root. Transforms operate in document coordinates and produce ops rather than mutating React state directly.

The transform system owns drag, resize, rotate, nudge, duplicate, constraints, snapping, alignment guides, spacing guides, and locked/hidden filtering.

Likely modules:

- `src/selection/selectionModel.ts`
- `src/geometry/transforms.ts`
- `src/geometry/hitTest.ts`
- `src/geometry/snapping.ts`
- `src/geometry/constraints.ts`

### Rendering Pipeline

SVG renders the editable surface because it is inspectable, deterministic, easy to map to scene nodes, good for selection overlays, and practical for hit-testing in a sample design tool. React renders panels and controls. Canvas can be introduced later for thumbnails, minimap, replay preview, or performance experiments without becoming the source of truth.

Rendering rules:

- Scene graph to memoized SVG node tree.
- Separate overlay layer for selection, handles, cursors, guides, pins, and prototype hotspots.
- Stable IDs in DOM attributes for Playwright and hit-test diagnostics.
- No renderer-specific state in the scene model.

Likely modules:

- `src/canvas/CanvasView.tsx`
- `src/canvas/SvgScene.tsx`
- `src/canvas/overlays/*`
- `src/render/nodeRenderers.tsx`

### Operations Model

All document changes are represented as typed operations. Each op includes `opId`, `sessionId`, `clientId`, `actorId`, `sequence`, `timestamp`, `kind`, and payload. Client-created ops are optimistic until the local server assigns canonical sequence numbers.

Undo and redo are built from inverse ops or reversible transactions. UI commands create transactions so compound actions, such as grouping or duplicating a selection, undo as one step.

Example op kinds:

- `node.create`
- `node.updateGeometry`
- `node.updateStyle`
- `node.reparent`
- `node.reorder`
- `node.delete`
- `component.create`
- `component.detachInstance`
- `comment.createThread`
- `prototype.linkFrame`
- `snapshot.create`

Likely modules:

- `src/ops/types.ts`
- `src/ops/apply.ts`
- `src/ops/invert.ts`
- `src/ops/journal.ts`
- `src/ops/transactions.ts`

### Replay System

Replay reconstructs a design from an initial snapshot plus the canonical ops journal sorted by server sequence, then stable op ID. Branching from a replay step creates a new version whose base snapshot is the reconstructed state at that step and whose journal starts from a branch marker.

Replay UI supports play, pause, step forward, step back, scrubber, speed controls, and branch from here. Determinism is verified by applying the same initial state and same journal twice and comparing canonical serialization.

Likely modules:

- `src/replay/replayEngine.ts`
- `src/replay/ReplayPanel.tsx`
- `src/replay/branching.ts`

### Sync Engine

The local Node server is authoritative for operation ordering. Clients send optimistic ops. The server validates basic shape, assigns canonical sequence numbers, persists to `data/sessions/<sessionId>.json`, and broadcasts ordered ops to connected clients.

Conflict handling:

- Prefer last-writer wins for scalar style and geometry fields.
- Respect explicit parent and z-order ops by canonical sequence.
- Reject ops that target missing nodes unless the op is idempotent or references a node created earlier in the same transaction.
- Keep presence, cursor, and selection messages ephemeral and out of the persistent journal.

Likely modules:

- `server/index.ts`
- `server/sessions.ts`
- `server/protocol.ts`
- `src/collab/client.ts`
- `src/collab/messages.ts`

### Export Pipeline

The canonical design JSON is the source of truth. Export first canonicalizes and validates the design, then generates a deterministic React + Tailwind component package. Component definitions become reusable React components; instances become component usage with override props where possible.

Export output:

- `index.ts`
- `Design.tsx`
- `components/*.tsx`
- `tokens.ts`
- `assets/*`
- optional `README.md`

Same design JSON must produce byte-stable generated output.

Likely modules:

- `src/serialization/canonical.ts`
- `src/export/codegen.ts`
- `src/export/files.ts`
- `scripts/export.ts`

## Risk Register

| Risk | Impact | Mitigation |
| --- | --- | --- |
| Rendering drift between model and SVG | Broken selection, export, or replay | Keep model renderer-independent; test scene-to-SVG snapshots for representative nodes. |
| Transform math bugs | Drag, resize, rotate, and constraints feel unreliable | Centralize geometry math and unit test transforms, constraints, and hit-testing. |
| Re-render storms during drag | Poor demo performance | Keep transient pointer state local, memoize render paths, batch op commits at interaction end where appropriate. |
| Realtime sync divergence | Two tabs disagree about design state | Server assigns sequence numbers; deterministic apply; tests replay same op stream on multiple clients. |
| Replay nondeterminism | Replay and version history lose credibility | Stable IDs, canonical ordering, canonical serialization, deterministic timestamps in tests. |
| Export output churn | Snapshot tests fail and generated code is hard to review | Stable sorting, formatting, component naming, and codegen snapshot tests. |
| Scope creep per milestone | Giant diffs and hard reviews | Implement one active milestone only; review diff against `docs/plans.md` before commit. |
| Overuse of dependencies | Core behavior becomes opaque | Use lean dependencies only for commodity concerns; own scene, ops, replay, sync protocol, and export logic. |
| Local persistence corruption | Demo files or sessions become unrecoverable | Write session files atomically; keep tracked demo seeds separate from gitignored runtime data. |
| UI polish crowding engineering work | Demo looks good but foundations are weak | Deterministic spine first; add polish continuously but never bypass verification gates. |

## Three-Minute Demo Script

1. Open Design Desk with `npm run dev`. The AI Builder landing page starter file loads immediately.
2. Show the professional editor shell: top toolbar, layers/assets panel, canvas, properties inspector, comments/replay/version panels.
3. Drag a hero card, resize text, snap to guides, duplicate a button, and show layers updating.
4. Switch to a second browser tab with the same local share link. Show presence, live cursors, remote selection outline, and an edit syncing back to the first tab.
5. Add a comment pin, reply in the thread, resolve it, then jump from the comments list back to the pin.
6. Open replay mode, scrub through the recent edits, step backward and forward, change speed, and branch from a prior point.
7. Open prototype preview, click a hotspot from the landing page to a dashboard frame.
8. Export JSON and React/Tailwind component package, then point to deterministic snapshot tests as the engineering proof.
